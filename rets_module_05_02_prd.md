## 1. 제품 개요

### 1.1. 문제 정의

소프트웨어 개발 프로젝트에서 요구사항과 테스트케이스 사이의 연결이 끊기면 다음의 문제가 발생한다.

- **검증 누락**: 요구사항에 대응하는 테스트가 없어 기능 결함이 출시 이후에 발견됨
- **테스트 낭비**: 요구사항과 무관한 테스트가 반복 수행되어 QA 공수 낭비
- **변경 관리 실패**: 요구사항이 변경될 때 영향받는 테스트케이스를 파악하지 못해 회귀 테스트 누락
- **추적 불가**: 요구사항과 테스트 결과 간 추적 관계가 없어 품질 현황 파악 곤란
- **수작업 의존**: 테스트케이스 초안 작성이 전적으로 수작업이어서 초기 품질이 담당자 역량에 좌우됨

**RETS-MODULE-05**는 이러한 문제를 해소하기 위해, 정의된 개발 요구사항(DEV-REQ)으로부터 LLM을 활용해 실행 가능한 테스트케이스를 자동 생성하고, 양방향(요구사항 ↔ 테스트케이스) 추적을 지원하는 도구이다.

### 1.2. 제품 비전

> "요구사항이 정의되는 순간, 검증 체계도 함께 만들어진다."

요구사항 기반 테스트케이스 생성 도구는 요구사항과 테스트를 하나의 흐름으로 연결하여, 검증 활동의 누락을 방지하고 테스트 품질을 체계화하는 사내 요구공학 도구 스위트(RETS)의 최종 검증 단계 모듈이다.

## 3. 유저 스토리 (RFP 기반 상세화)

### US-01. LLM 기반 테스트 시나리오·케이스 자동 생성 및 검토

**As** QA 엔지니어로서,
**I want** 개발요구사항 목록을 불러오면 LLM이 각 요구사항에 대해 정상·예외·경계 시나리오와 Given-When-Then 형식의 테스트케이스를 자동으로 제안해 주기를 바라며,
**So that** 테스트케이스 초안 작성 시간을 줄이고 누락 없는 검증 체계를 빠르게 구성할 수 있다.

**수락 기준 (Acceptance Criteria)**:
- DEV-REQ JSON 파일을 불러오면 요구사항 목록이 파싱되어 화면에 표시된다
- 요구사항 단위로 LLM 호출을 실행하면 정상/예외/경계 3개 이상의 시나리오와 Given-When-Then 형식의 테스트케이스가 반환된다
- 반환된 각 테스트케이스에 대해 Accept / Reject 버튼이 제공된다
- Accept된 테스트케이스는 테스트케이스 목록에 등록되고 요구사항과 매핑된다

---

### US-02. 테스트케이스 수동 편집·보강 및 추적 관리

**As** QA 엔지니어로서,
**I want** LLM이 생성한 테스트케이스를 직접 수정·추가·삭제하고, 요구사항과의 매핑 관계를 편집할 수 있기를 바라며,
**So that** 자동 생성 결과를 실무 맥락에 맞게 보완하고 정확한 추적 관계를 유지할 수 있다.

**수락 기준**:
- 테스트케이스 목록에서 제목, 전제조건, 입력값, 절차, 예상결과를 인라인 편집 가능하다
- 새 테스트케이스를 수동으로 추가할 수 있다
- 기존 테스트케이스를 삭제할 수 있다
- 요구사항↔테스트케이스 매핑을 추가/제거할 수 있다

---

### US-03. 변경 영향 분석 및 LLM 기반 테스트케이스 변경 가이드

**As** 요구사항 분석가로서,
**I want** 특정 요구사항이 변경되었을 때 영향받는 테스트케이스를 즉시 확인하고, LLM이 각 케이스에 대해 '유지 / 수정 / 폐기 / 신규 추가' 권고를 제시해 주기를 바라며,
**So that** 요구사항 변경으로 인한 테스트케이스 누락이나 불일치를 체계적으로 관리할 수 있다.

**수락 기준**:
- 요구사항 변경 표시(변경 플래그) 설정 시 연관된 테스트케이스가 자동 하이라이팅된다
- LLM 호출을 통해 각 영향 테스트케이스에 대한 권고(유지/수정/폐기/신규) 및 근거가 표시된다
- 권고 내용을 수락하거나 거부할 수 있다

---

### US-04. 양방향 추적성 그래프 시각화

**As** PM으로서,
**I want** 요구사항·시나리오·테스트케이스 간의 연결 관계를 네트워크 그래프로 시각적으로 확인하고 싶으며,
**So that** 어떤 요구사항이 검증되었고 어떤 요구사항에 테스트가 없는지 한눈에 파악할 수 있다.

**수락 기준**:
- 요구사항, 시나리오, 테스트케이스를 노드로, 매핑 관계를 엣지로 표현하는 그래프가 표시된다
- 노드 클릭 시 해당 항목과 연결된 노드만 강조(포커스 모드)된다
- 팬(Pan) 및 줌(Zoom) 인터랙션이 지원된다
- 요구사항 유형 및 테스트 유형 기준 클러스터링을 제공한다

---

### US-05. 테스트케이스 목록 및 추적 매트릭스 파일 출력

**As** QA 엔지니어로서,
**I want** 완성된 테스트케이스 목록과 요구사항↔테스트케이스 추적 매트릭스를 CSV 파일로 내보낼 수 있기를 바라며,
**So that** 외부 도구(Excel, 테스트 관리 시스템 등)에서 재활용하거나 산출물로 제출할 수 있다.

**수락 기준**:
- [테스트케이스 내보내기] 버튼 클릭 시 CSV 파일이 다운로드된다
- [추적 매트릭스 내보내기] 버튼 클릭 시 요구사항 ID ↔ 테스트케이스 ID 매핑이 담긴 CSV 파일이 다운로드된다
- 출력 CSV는 자체 정의 스키마를 준수한다

- RETS 공통 디자인 시스템 (`rets_02b_design_requirements.md`) 준수

- 에러 상태(파싱 실패, API 오류)에 대한 명확한 피드백 메시지 제공

## 6. 기술 고려사항

### 6.1. 구현 기술 스택

| 항목 | 내용 |
|------|------|
| 구현 언어 | Vanilla JavaScript (ES6+) |
| 구현 형태 | 단일 HTML 파일 (CSS·JS 내장) |
| LLM 연동 | LLM 릴레이 서버 (ai-relay.mooo.com), HTTPS, team_id: team2 |
| 그래프 시각화 | D3.js 또는 경량 네트워크 그래프 라이브러리 (CDN 활용) |
| 데이터 저장 | 세션 메모리(In-Memory) 기반, 브라우저 새로고침 시 초기화 |
| 파일 I/O | 입력: FileReader API, 출력: Blob + downloadURL |

### 6.2. LLM 연동 방식

- 채팅 세션 생성(Initialize Chat with System Prompt) → 이후 메시지 전송(Continue Chat) 흐름
- Request body에 `team_id: "team2"` 및 구조화된 프롬프트(JSON 반환 요청 포함) 전달
- Response의 텍스트에서 JSON 파싱, 유효성 검사 후 UI에 반영
- 파싱 실패 시 오류 메시지 표시 + 재시도 버튼 제공
- 시스템 프롬프트 참조: `/00_rfp_requirements/rets_02a_llm_system_prompt.md`

### 6.3. 입출력 스키마

**입력**
- 형식: JSON
- 스키마: `rets_02c_req_schema.json` (DEV 타입 req_type 필터링)

**출력 — 테스트케이스 목록 CSV**
- 자체 정의 스키마 (최소 포함 컬럼): `tc_id`, `req_id`, `scenario_type`, `title`, `precondition`, `input`, `steps`, `expected_result`, `status`

**출력 — 추적 매트릭스 CSV**
- 컬럼: `req_id`, `req_name`, `tc_id`, `tc_title`, `mapping_type`

### 6.4. 데이터 흐름

```
[JSON 파일 업로드]
        ↓
[DEV-REQ 파싱 및 목록 표시]
        ↓
[요구사항 선택 → LLM 호출 (시나리오+테스트케이스 생성)]
        ↓
[사용자 Accept/Reject/편집]
        ↓
[테스트케이스 목록 + 추적 매트릭스 업데이트]
        ↓
[그래프 시각화 업데이트] ←→ [변경 영향 분석 (요구사항 변경 표시 시)]
        ↓
[CSV 파일 출력]
```
