## 1. 개요

### 1.1. 목적

본 문서는 `rets_module_05_03_srs.xlsx`에 정의된 소프트웨어 요구사항을 기반으로, RETS-MODULE-05의 검증에 필요한 **테스트 시나리오**와 **테스트 케이스**를 설계한다. 각 테스트 케이스는 SRS의 요구사항 ID와 양방향 추적 관계를 유지한다.

## 2. 핵심 사용자 시나리오 (Key User Scenarios)

### KUS-01. DEV-REQ 로드 → LLM 테스트케이스 생성 → 검토 → CSV 출력

> 가장 핵심적인 End-to-End 플로우. Playwright 자동화 대상.

**시나리오 목표**: 사용자가 DEV-REQ JSON을 업로드하고, LLM 기반으로 테스트케이스를 생성·수락·편집한 뒤 CSV로 출력하는 전체 흐름이 정상 동작한다.

**전제조건**:
- `rets_benchmark_02_dev_req.json` 파일이 준비되어 있다
- 설정 화면에서 team_id = "team2"가 입력되어 있다
- LLM 릴레이 서버가 정상 동작 중이다

**절차**:
1. 앱을 브라우저로 열고 설정에서 team_id "team2" 입력
2. [JSON 불러오기] 버튼 클릭 → `rets_benchmark_02_dev_req.json` 선택
3. 요구사항 목록 그리드에 5건 이상의 DEV-REQ가 표시되는지 확인
4. 요구사항 1건 선택 → [테스트케이스 생성] 버튼 클릭
5. LLM 로딩 스피너 표시 확인 후 결과 대기 (최대 30초)
6. 생성된 테스트케이스 목록에서 정상/예외/경계 시나리오 각 1건 이상 확인
7. 최소 1건 Accept, 1건 Reject 수행
8. Accept된 TC에 대해 예상결과 필드 인라인 편집
9. [TC CSV 내보내기] 클릭 → 파일 다운로드 확인
10. [추적 매트릭스 CSV] 클릭 → 파일 다운로드 확인

**예상 결과**: 모든 단계 정상 완료, 다운로드된 CSV에 Accept된 TC 데이터 포함

---

### KUS-02. 변경 요구사항 영향 분석 → LLM 권고 → 수락/거부

**전제조건**: KUS-01 수행 완료 상태 (TC 및 매핑 존재)

**절차**:
1. 요구사항 목록에서 임의 요구사항에 변경 플래그 설정
2. 추적 매트릭스에서 해당 요구사항 연관 TC 하이라이팅 확인
3. [변경 분석 실행] 버튼 클릭
4. LLM 권고 결과(유지/수정/폐기/신규) 각 TC별 확인
5. 1건 수락, 1건 거부 후 TC 상태 변경 확인

**예상 결과**: 권고 결과 수락/거부가 TC 상태에 반영

---

### KUS-03. 추적성 그래프 시각화 탐색

**전제조건**: KUS-01 수행 완료 상태

**절차**:
1. [그래프 시각화] 탭 이동
2. 요구사항·TC 노드가 그래프로 표시되는지 확인
3. 요구사항 노드 클릭 → 연결된 노드만 강조 확인
4. 팬(마우스 드래그) 및 줌(휠 스크롤) 동작 확인

**예상 결과**: 그래프 렌더링 정상, 인터랙션 동작

### TS-001 | DEV_FR_001_01 — JSON 파일 업로드 및 파싱

#### TS-001 [정상] 유효한 DEV-REQ JSON 파일 업로드

| 항목 | 내용 |
|------|------|
| SRS 추적 | DEV_FR_001_01 |
| 시나리오 유형 | 정상 |
| 우선순위 | MUST |
| 자동화 | Playwright |

**Given**: 앱이 실행 중이고 유효한 DEV-REQ JSON 파일(`rets_benchmark_02_dev_req.json`)이 준비되어 있다

- 상단에 "X건 로드 완료" 또는 유사한 성공 메시지가 표시된다
