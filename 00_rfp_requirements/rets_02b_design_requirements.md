## RFP-UI-REQ-01. 시각 스타일 원칙

- 전체 UI는 깔끔하고 현대적인 사용자 느낌(예: shadcn/ui 계열 스타일 참고)을 바탕으로 미니멀하고 정제된 업무형 인터페이스로 구현되어야 한다.
  - 일관된 컴포넌트 체계, 충분한 가독성, 단순한 시각 언어, 과도하지 않은 장식 요소를 유지한다.

- 색상 체계는 기본/보조/강조/상태/비활성 상태를 구분하여 정의하여야 하며, 라이트/다크 모드 모두에서 텍스트와 배경 간 충분한 가독성을 확보하여야 한다. 특히 `disabled`, `read-only`, `selected`, `hover`, `focus` 상태는 명확히 식별 가능해야 한다.
  - 기본 구조색은 무채색 중심으로 구성하며, **포인트 컬러(`#F97316`)** 를 강조색으로 사용하여 UX를 향상해야 한다. 그라데이션은 사용하지 않는다.
  - 포인트 컬러 혹은 그 variation(예: `#FB923C`)을 통해 Hover, Active, Selected, Badge, Progress Emphasis 등에 활용한다.
  - 오류, 경고, 성공, 정보 상태는 별도의 의미색 체계를 사용한다. 단, 포인트 컬러 수준의 탁도를 가지는 색상을 사용하여 시각적 피로감을 유발하지 않도록 한다. 라이트/다크 테마에 따라 적절한 가독성에 맞는 색상을 사용한다.

- **의미색 체계 (공통 정의 — `design_system.html` 참고)**

  | 상태 | 용도 | 라이트 메인 | 라이트 배경 | 라이트 보더 | 다크 메인 | 다크 배경 | 다크 보더 |
  |------|------|------------|------------|------------|----------|----------|----------|
  | Error (오류) | 필수값 누락, 처리 실패, 유효성 오류 | `#8c3131` | `#FEF2F2` | `#FECACA` | `#F87171` | `#450A0A` | `#991B1B` |
  | Warning (경고) | 주의 필요, 불완전 상태, 검토 권고 | `#D97706` | `#FFFBEB` | `#FDE68A` | `#FBBF24` | `#451A03` | `#92400E` |
  | Success (성공) | 처리 완료, 검증 통과, 저장 성공 | `#1CB357` | `#F0FDF4` | `#BBF7D0` | `#3DD171` | `#052E16` | `#166534` |
  | Info (정보) | 안내, 힌트, 비중립 참고 사항 | `#3375E8` | `#EFF6FF` | `#BFDBFE` | `#5299F0` | `#172554` | `#1E40AF` |

  > Success와 Info의 메인 색상은 기존 순색(`#22C55E`, `#3B82F6`)보다 채도를 낮추고 명도를 줄여 장시간 사용 시 시각적 피로감을 최소화한 값으로 조정하였다.

- **중립색 체계 (Neutral Colors)**

  | 토큰 | 라이트 | 다크 | 용도 |
  |------|--------|------|------|
  | `--bg` | `#FFFFFF` | `#101111` | 전체 배경 |
  | `--surface` | `#F9FAFB` | `#1c1d1d` | 카드·패널 배경 |
  | `--surface-2` | `#F3F4F6` | `#262727` | 내부 구분 영역 |
  | `--border` | `#E5E7EB` | `#303232` | 기본 테두리 |
  | `--border-md` | `#D1D5DB` | `#404242` | 강조 테두리 |
  | `--text-pri` | `#111827` | `#F8FAFC` | 본문 텍스트 |
  | `--text-sec` | `#6B7280` | `#94A3B8` | 보조 텍스트 |
  | `--text-dis` | `#9CA3AF` | `#475569` | 비활성 텍스트 |

  - 입력 필드와 주요 상호작용 요소는 키보드 탐색이 가능해야 하며, `focus` 상태가 명확하게 식별되어야 한다.

- 기본 테마는 **라이트 모드**이며, 다크 모드 전환 버튼을 통해 테마 변경이 가능해야 한다.
  - 다크 모드 배경색은 슬레이트 계열이 아닌 **웜-다크 계열(`#101111`)** 을 사용한다.

- 폰트는 **`Noto Sans KR`** 만을 사용해야 하며, 텍스트 사이즈는 **20px을 절대 넘지 않도록** 한다. 자간(letter-spacing)은 **-0.03em ~ -0.05em** 범위 내에서 적용하며, 기본값은 **-0.04em**으로 한다.

  **타이포그래피 스케일**

  | 역할 | 크기 | 굵기 | 자간 | 용도 |
  |------|------|------|------|------|
  | Title | 20px | Bold (700) | -0.04em | 모듈 제목, 페이지 헤딩 |
  | Subtitle | 16px | Bold (700) | -0.04em | 섹션 제목, 그룹 헤딩 |
  | Body | 14px | Regular (400) | -0.04em | 본문, 설명 텍스트 |
  | Label | 14px | Bold (700) | -0.04em | 강조 레이블 |
  | UI Label | 13px | Medium (500) | -0.04em | 버튼, 탭, 입력 레이블 |
  | Caption | 12px | Regular (400) | -0.03em | 힌트, 도움말, 캡션 |
  | Meta | 11px | Regular (400) | -0.03em | 뱃지, 타임스탬프, 메타정보 |

  **인라인 코드 스타일 (`<code>` 태그)**

  - `<code>` 태그를 사용하는 인라인 코드(요구사항 ID, 상태값, 정규식, 컬러 코드 등)에는 별도 스타일을 적용한다.

  | 속성 | 값 | 비고 |
  |------|-----|------|
  | 폰트 | `Consolas, 'Courier New', monospace` | Noto Sans KR 예외 허용 영역 |
  | 크기 | `0.75em` (본문 14px 기준 약 10.5px) | 본문보다 작게 유지 |
  | 색상 | `#374151` (라이트) / `#9CA3AF` (다크) | 진한 회색 계열 |
  | 배경 | `var(--surface-2)` | 본문 배경과 구분 |
  | 패딩 | `1px 5px` | — |
  | 자간 | `0` | 본문의 `-0.04em` 리셋 |

  - 필요시 굵은체(bold) 사용 가능
  - 주요 액션은 버튼 형태로 명확하게 제공하여야 한다. 보조 탐색이나 부가 정보 접근 성격의 액션은 링크 스타일 UI를 제한적으로 사용할 수 있다.

- 불필요한 그림자, 애니메이션은 사용하지 않는다.

- 상태, 태그, 뱃지 등의 UI는 불필요한 줄바꿈이 발생하지 않도록 처리한다.

## RFP-UI-REQ-02. 공통 UI 구성 원칙

- 모듈의 HTML 파일을 브라우저에서 열었을 경우 **파비콘과 제목**이 정상적으로 표시되어야 한다.

- 모듈의 업무 화면은 **Header – (Working Tabs) – Body** 구조로 구성된다.

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ [모듈 아이콘(선택)] [모듈 제목]     [조직 로고]     [◐] [불러오기] [내보내기] │
└──────────────────────────────────────────────────────────────────────────────┘
┌──────────────┬──────────────┬──────────────┬──────────────┐
│ Tab 1        │ Tab 2        │ Tab 3        │ Tab 4        │ (선택사항)
└──────────────┴──────────────┴──────────────┴──────────────┘
┌──────────────────────────┊──────────────────────────────────┐
│ Column A                 ┊ Column B (필요시에만 2-column)    │
│                          ┊                                   │
│ 주요 입력 / 목록 / 편집  ┊ 분석 결과 / 미리보기 / 설정 영역  │
│                          ┊                                   │
└──────────────────────────┊──────────────────────────────────┘
                           ↑ 드래그로 너비 조절 가능
```

### Header 구성

- 업무 화면 최상단에는 모듈 제목과 데이터 컨트롤 버튼 그룹이 위치한 **Header**가 존재해야 한다.

  **Header 레이아웃 — 3-column 구조**

  | 영역 | 위치 | 내용 |
  |------|------|------|
  | 좌측 (`flex: 1`) | 왼쪽 정렬 | 모듈 뱃지 + 모듈명 |
  | 중앙 (`flex-shrink: 0`) | 중앙 고정 | 조직 로고 이미지 |
  | 우측 (`flex: 1`) | 오른쪽 정렬 | 버튼 그룹 (테마 전환, 불러오기, 내보내기 등) |

  **Header 세부 규칙**
  - 모듈 이름: 한글, 좌측 정렬
  - 조직 로고: 헤더 **중앙** 고정 배치. 라이트/다크 모드에 따라 별도 이미지 파일(`logo_light.png` / `logo_dark.png`)을 CSS로 전환하여 사용한다.
  - 버튼 그룹: 우측 정렬
    - **테마 변경**: 아이콘 혹은 토글 형식으로 표시 (라이트/다크모드)
    - **불러오기**: 지정된 입력 파일(벤치마크 요구사항) 로드
    - **내보내기**: 모듈별 지정된 출력 형식으로 Export (1개 형식 이상 가능)
  - 헤더는 Body의 스크롤과 무관하게 최상단 고정

  **Header 시각 스타일**
  - 헤더 배경색은 Body 배경(`--bg`)과 구별되는 별도 색상(`--header-bg`)을 사용한다.
    - 라이트 모드: `rgba(240, 242, 245, 0.72)` (연회색 계열, 반투명)
    - 다크 모드: `rgba(14, 35, 35, 0.80)` (어두운 청록색 계열, 반투명)
  - 헤더는 **Blur Glass** 효과를 적용하여 스크롤 시 뒤 콘텐츠가 흐릿하게 비쳐 보이도록 한다.
    - `backdrop-filter: blur(14px) saturate(160%)` 적용
    - 배경 반투명(`rgba`) 처리가 선행되어야 효과가 표현된다.
  - 모듈 구분 뱃지(예: `RETS`)의 배경색은 포인트 컬러 Pressed 톤(`#C2560B`)을 사용한다.

### Working Tabs (선택사항)

- 모듈 내의 기능을 구분하기 위해 탭 구성이 필요할 경우, Header와 Body 사이에 Working Tabs를 구성하여 여러 기능 화면을 제공할 수 있다.

### Body

- Body는 최대 **2-column**으로 구성할 수 있으며, 반응형 웹 원칙(RFP-UI-REQ-05)에 따라 뷰포트 크기가 작아졌을 때 레이아웃이 자동 조정되어야 한다.
- 2-column 구성 시, 두 컬럼 사이에 **드래그로 너비를 조절할 수 있는 리사이저(Resizer)** 를 제공하여야 한다.
  - 리사이저는 호버 및 드래그 중 포인트 컬러로 강조 표시되어야 한다.
  - 각 컬럼의 최소 너비는 160px 이상을 유지하여 콘텐츠 가독성을 확보한다.

## RFP-UI-REQ-03. 공통 UI 컴포넌트 원칙

- 모든 입력 컴포넌트는 사용 목적과 입력 맥락이 명확히 드러나야 하며, 필요한 경우 `placeholder`, `help text`, 예시값, 단위 표시, 검증 메시지 등을 제공하여야 한다.

- 입력 필드는 다음 상태를 명확히 구분하여 시각적으로 표현하여야 한다.

  | 상태 | 설명 | 라이트 보더 | 다크 보더 |
  |------|------|------------|----------|
  | Default | 기본 입력 가능 상태 | `#D1D5DB` | `#404242` |
  | Focus | 포커스 상태 (키보드 탐색 포함) | `#F97316` (포인트 컬러) | `#F97316` |
  | Error (오류) | 유효성 실패, 필수값 누락 | `#8c3131` | `#92400E` |
  | Warning (경고) | 주의 필요, 중복 가능성 등 | `#451A03` | `#92400E` |
  | Disabled | 수정 불가 비활성 상태 | `#E5E7EB` | `#303232` |
  | Read-only | 표시 전용, 편집 불가 | `#E5E7EB` | `#303232` |

  - Focus 상태는 포인트 컬러 3px outer glow(`box-shadow: 0 0 0 3px var(--brand-50)`)를 함께 표시한다.
  - 경고(Warning) 상태의 보더 색상은 라이트 모드에서 `#451A03`을 사용한다.

- 사용자의 오입력 가능성이 높거나, 의미 해석이 어려운 항목, 전문 용어가 포함된 항목에 대해서는 **툴팁 또는 보조 설명**을 제공하여야 한다.

- 모든 UI 컴포넌트에 대해 **2026년 상반기 기준 AI Slop 등의 안티패턴**을 포함하지 않도록 구현한다.
  - 참고: [https://impeccable.style/slop/](https://impeccable.style/slop/)

- 데이터와 관련된 컴포넌트(예: 그리드)는 다음 대표 상태에 대한 UI를 포함하여야 한다:
  - 데이터 없음 (Empty)
  - 로딩 (Loading)
  - 오류 (Error)
  - 권한 없음 (Forbidden)

- 업무 화면 내 **팝업은 일절 사용하지 않으며**, 인터랙션 성격에 따라 **모달(Modal)** 또는 **드로어(Drawer)** 로 구현한다.

  **모달과 드로어의 사용 기준**

  | 구분 | 모달 (Modal) | 드로어 (Drawer) |
  |------|-------------|----------------|
  | 주요 용도 | 확인/취소가 필요한 액션, 되돌릴 수 없는 작업 경고, 간단한 설정 변경, 짧은 폼 (1~3개 필드) | 항목 상세 보기, 컨텍스트 유지 편집, 긴 폼·다중 필드 입력, 메타데이터·필터 패널 |
  | 레이아웃 | 화면 중앙 오버레이 | 화면 우측 슬라이드인 패널 |
  | 너비 | 최대 460px | 최대 400px (max 90vw) |
  | 구조 | Header – Body – Footer | Header – Body (스크롤) – Footer |

  **공통 구현 규칙 (모달·드로어 모두 적용)**
  - 영역 간 구분선(border) 사용 금지 (단, 드로어 Footer 상단은 예외)
  - 닫기(×) 버튼은 **테두리 없이** 아이콘 형태로만 표시
  - 오버레이(배경 어둠 처리) 영역 클릭으로 닫기 지원
  - 오버레이 배경색: `rgba(0,0,0,0.45)`

  **드로어 추가 규칙**
  - 드로어는 화면 우측에서 슬라이드인(`translateX`) 방식으로 진입하며, 열림/닫힘 애니메이션은 `0.25s ease`를 적용한다.
  - 드로어가 열려 있는 동안 배경 스크롤을 비활성화(`overflow: hidden`)하여 포커스를 유지한다.
  - 드로어 Body 영역은 콘텐츠가 길어질 경우 독립적으로 스크롤되어야 하며, Header와 Footer는 고정 위치를 유지한다.
  - 드로어 왼쪽 경계에는 `border-left: 1px solid var(--border)`를 적용하여 본문과 시각적으로 분리한다.

- 기본 `Alert`을 절대 사용하지 말고, **우측 하단의 Toast**를 출력하도록 한다.
  - Toast는 단순 Validation/정보 제공 외에도 버튼이 포함될 수 있으며, 버튼이 포함된 경우 사용자 입력 전까지 자동으로 사라지지 않도록 한다.
  - Toast는 상태 유형별로 구분되어야 하며, 동일 메시지의 과도한 중복 노출을 방지하여야 한다.
  - 동시에 과도한 수의 Toast가 누적되어 주요 업무 화면을 방해하지 않도록 제어하여야 한다. 최대 동시 표시 개수는 **4개**를 초과하지 않도록 한다.

- 디자인 시스템 내 컬러 코드(`#RRGGBB` 형식) 표시 요소는 다음 규칙을 따른다.
  - 컬러 코드 텍스트에는 `Consolas, 'Courier New', monospace` 폰트를 적용한다.
  - 컬러 코드 요소는 클릭 시 해당 값이 클립보드에 복사되어야 한다.
  - 복사 완료 시 우측 하단 Toast(`success` 타입)로 복사된 코드값과 함께 안내 메시지를 표시한다.
  - 동일 코드를 연속 클릭 시 이전 복사 토스트를 교체하여 중복 누적을 방지한다.

## RFP-UI-REQ-04. 업무화면 · 기능 · AI UX 원칙

- 각 모듈별로 **핵심 사용자 시나리오를 관통하거나 모니터링하는 목록화면/대시보드**가 진입 화면이며, 해당 화면의 구성은 화면설계서를 따른다.

- 대량 반영, 삭제, 덮어쓰기, 승인 등 **되돌릴 수 없는 작업**에 대해서는 사용자 확인 절차를 제공하여야 한다.

- 비동기 기능이 수행되는 경우, 사용자는 처리 중 상태를 명확히 인지할 수 있어야 하며, **진행 상태(스피너 등), 성공, 실패 여부**를 확인할 수 있어야 한다.
  - File I/O가 발생하는 부분에 대해 비동기 처리 UI가 지원되어야 한다.
  - LLM을 사용한 AI Call 처리는 외부 서버를 통해 비동기로 동작하므로, 비동기 처리 UI가 지원되어야 한다.

- AI(LLM)로부터 반환된 값은 **자동으로 반영되어서는 안 되며**, 사용자가 개입하여 **수락(Accept) / 거부(Reject)** 할 수 있는 UI가 제공되어야 한다.

## RFP-UI-REQ-05. 반응형 웹 원칙

- 본 모듈은 **데스크톱 환경(기본 1280px 이상)** 을 우선 기준으로 설계한다. 다만 viewport 축소 시 주요 기능이 손실되지 않도록 레이아웃 재배치, 영역 축소, 스크롤 처리, 정보 우선순위 조정 등의 반응형 대응을 제공하여야 한다.

- 모바일 전용 UX 최적화는 필수 범위가 아니나, **저해상도 환경에서도 사용 불가능한 상태가 되어서는 안 된다**.

- 콘텐츠 영역은 항상 **내부 30px의 패딩**을 가진 viewport 전체 width를 최대폭으로 사용한다.
